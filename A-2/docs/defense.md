# A-2 答辩讲稿

## 1. 模块定位

A-2 是可视化 ATC 地空通话语音标注系统中的语音处理模块，主要负责把 LiveATC 的实时流和历史音频接入系统，生成时间戳，完成本地存储管理，并对外提供按时间范围查询和导出语音的服务接口。

如果用一句话概括，本模块解决的是：

“把原始地空通话音频变成可按任意时间范围检索、下载和切片的结构化语音数据。”

## 2. 本次实现的四个核心目标

对应课程要求，本项目已经完成以下四条主线：

1. 以网络数据流的形式接收 LiveATC 服务器的地空通话数据，并生成时间戳。
2. 下载 LiveATC 服务器的历史地空通话数据，并生成时间戳。
3. 实现语音数据的本地存储管理。
4. 实现服务 API 接口，按起止时间获取语音数据。

## 3. 设计思路

本模块没有把整段长音频作为唯一操作对象，而是采用“分段存储、按需重组”的思路：

1. 实时流或历史音频进入系统后，先写入本地文件系统。
2. 同时把 ICAO、频段、时间范围、文件路径、大小、校验值等元信息写入数据库。
3. 查询时不要求和原始分段完全对齐，而是通过时间重叠规则找到命中的多个分段。
4. 如果用户需要一段完整音频，就对命中的分段进行裁剪和拼接，再生成最终文件返回。

这样做的好处是：

- 存储层结构清晰，便于管理。
- 查询层灵活，能支持任意时间段检索。
- 接口层既能返回元信息，也能直接导出音频文件。

## 4. 演示顺序建议

### 第一步：检查服务状态

说明系统已经启动、数据库已经初始化、后台同步线程已经工作。

### 第二步：按时间范围查询语音分段

说明查询不是“按文件名查”，而是“按时间窗口查重叠分段”。

### 第三步：直接导出一段指定时间范围的完整音频

说明系统会自动完成：

- 查询重叠分段
- 逐段裁剪
- 顺序拼接
- 回传最终音频

### 第四步：展示元信息同步能力

说明系统不仅能存，还能校验和修正元信息，并且在文件缺失时返回明确错误。

### 第五步：展示实时接收能力

说明实时流会按接收时间切段保存，并可观察运行状态、最新分段时间和异常信息。

## 6. 目前仍然存在的不足

这部分建议主动讲出来，反而会显得项目边界清晰：

1. 时间戳语义不是完全等于上游源端原始时间。
   实时流时间戳是系统接收和切段时间；历史文件时间戳主要来自文件名起始时间和音频时长推算。

2. 可靠性能力是课程项目级，不是完整生产级。
   目前已经有自动重连、基础断点续传和同步修复，但还没有完整的任务调度器和恢复编排。

3. 部分上游能力依赖外部环境。
   例如 `ffmpeg/ffprobe` 影响长音频截断和混合格式拼接能力。

4. 还有进一步完善空间。
   例如孤儿文件自动补录、更多异常路径测试、更加完整的调度与告警能力。

## 7. 老师可能会问的问题

### 为什么要分段存储，而不是整段存储？

因为查询需求是“按任意起止时间取音频”，分段存储可以降低落盘压力，也更方便后续按时间重组。

### 为什么查询条件是时间重叠，而不是时间完全相等？

因为用户请求的时间范围通常不会正好等于原始分段边界，所以要用重叠匹配才能命中所有相关片段。

### 实时音频和历史音频是如何统一处理的？

两者最终都会落成文件并写入统一的 `a2_voice_info` 元信息表，因此查询、导出和下载接口都可以复用。

### 这个项目最有价值的技术点是什么？

最核心的是把“分段存储”和“任意时间范围重组”这条链路打通了，使系统可以既高效存储，又灵活对外提供语音服务。
