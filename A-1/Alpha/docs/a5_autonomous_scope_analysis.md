# A-5 自主可控模块分析与任务落地计划

## 1. 结论摘要

基于以下材料综合判断：

- `功能设计说明书第8组.docx`
- `可行性分析第8组修改版.docx`
- `数据库设计说明书 (2) (1).docx`
- `需求规格说明书第8组修改版.docx`
- `docs/a5_db_middleware_plan.md`

本次按 “A-5 主控口径” 对模块归属进行分析。所谓“完全依赖于我自己”，这里定义为：

- A-5 可以独立决定该模块的业务边界、接口形态和表结构
- 该模块即使需要服务其他模块，其核心字段定义权和实现节奏仍在 A-5
- 不依赖 A-1/A-2/A-3/A-4 先给出最终业务规则才能开始设计

综合现有文档，A-5 中最适合归入“完全自主”的部分主要是：

- 用户与鉴权能力
- VSP/AIP 辅助数据管理能力
- 消息中间件基础封装、事件规范、失败处理和监控
- A-5 自己维护的审计、日志、死信、系统支撑类表

需要强调的是，A-5 虽然承担“数据库接口与 VSP 数据”模块，但并不等于整个系统所有业务表都由 A-5 单方决定。文档中多处已经体现出 A-1/A-2/A-3/A-4 对航迹、语音、ASR、标注等数据模型拥有明显主导权，A-5 更适合承担统一接入、兼容存储、接口编排和平台治理职责。

## 2. A-5 模块职责边界总览

现有文档对 A-5 的一级功能范围基本一致，覆盖以下 5 类：

- 数据库运维与管理
- 跨模块 API 接口服务
- 用户管理与鉴权
- VSP/AIP 数据管理
- 消息中间件服务

但这 5 类内部并不是同一种“控制权”。按控制权来源，建议拆成三层：

### 2.1 完全由 A-5 主控

- 用户登录、Token 派发、刷新、注销、请求鉴权拦截
- 用户表、用户状态、密码策略、登录审计等支撑数据
- VSP/AIP 数据采集后的标准化入库模型
- VSP 查询接口、GeoJSON 输出接口
- Redis 队列命名规范、统一消息结构、失败重试、死信处理、消费监控
- A-5 自己新增的日志表、失败事件表、消费异常表、系统配置类表

### 2.2 A-5 主导接入，但需要跨模块协商

- 航迹数据推送接口
- 语音元数据接收/查询接口
- ASR 结果接收接口
- 标注数据加载接口
- 标注结果保存接口
- 航迹、语音、ASR、标注结果等跨模块共享表的最小兼容字段

这一层的特点是：A-5 负责“落库与对外承接”，但字段定义权不完全在 A-5，必须与数据生产方或主要使用方协商。

### 2.3 主要受外部/他模块约束

- ADS-B 航迹原始字段结构
- A-2 语音采集任务、下载任务、录音元数据结构
- A-3 输出的 ASR/VAD 结果结构
- A-4 标注任务流和标注结果字段形态
- 来自 AIP、VSP 原始资料的内容完整性与格式差异

这一层不适合由 A-5 单方“拍板”。A-5 可以定义接入方式、校验规则和兼容落库方案，但不宜把业务语义写死。

## 3. 模块归属判断表

| 模块/对象 | 归类 | 判断 | 依据 |
| --- | --- | --- | --- |
| 数据库运行环境 | 完全自主 | A-5 负责 SQLite/备份/部署策略/目录组织 | 需求与功能设计均将数据库运行维护列入 A5-DB |
| 用户管理与鉴权 | 完全自主 | 角色、认证、密码、Token、审计由 A-5 直接定义 | 需求与功能设计均将 USR 单列为 A-5 核心模块 |
| VSP/AIP 数据管理 | 完全自主 | A-5 负责采集、清洗、建模、查询输出 | 需求与功能设计均将 VSP 单列为独立模块 |
| 消息中间件基础服务 | 完全自主 | 队列规范、事件封装、失败策略、监控应由 A-5 统一定义 | 需求与功能设计均将 MSG 单列为独立模块 |
| 航迹数据推送接口 | 协商接入 | A-5 负责接收和存储，但字段语义受 A-1/数据源约束 | 文档明确 A-1 为航迹处理模块 |
| 语音元数据接口 | 协商接入 | A-5 负责接入与查询，但数据结构明显受 A-2 主导 | 数据库设计说明书明确 a2_* 由 A-2 主导 |
| ASR 结果接口 | 协商接入 | A-5 承接存储，但结果结构受 A-3 产出方式约束 | 可行性分析说明 A-3 负责识别结果生成 |
| 标注加载/保存接口 | 协商接入 | A-5 可提供接口，但任务流和字段需与 A-4 协调 | A-4 为标注界面主要使用方 |
| ADS-B 原始航迹模型 | 他方主导 | A-5 不应独立定义完整业务语义 | 数据来源和采集逻辑在 A-1 |
| a2_task_* 任务配置 | 他方主导 | 任务调度、下载、采集配置明显属于 A-2 内部模型 | 数据库设计说明书明确归 A-2 |
| annotation_* 全量模型 | 他方主导/协商 | 如仅承载保存可由 A-5 接库，但业务流程应与 A-4 定义一致 | 标注流程本质由 UI/业务流程驱动 |

## 4. 完全自主模块清单

## 4.1 用户与鉴权模块

这是 A-5 中最明确的自主模块之一。现有文档已经把以下能力归入 A-5 的直接职责：

- 登录
- Token 派发
- Token 刷新
- 注销
- 鉴权拦截
- 角色权限定义
- 密码安全策略

### API 设计权判断

这部分 API 的路径、入参、响应结构、错误码、JWT 生命周期、刷新规则都可以由 A-5 主导，只需要给 A-4 提供清晰契约。

### 表设计权判断

以下表应视为 A-5 完全主控：

- `users`
- `user_login_audit`
- `user_tokens` 或刷新令牌表
- 轻量角色表，或第一阶段直接在 `users.role` 中落地

### 结论

这是优先级最高、最适合先实现的自主部分。

## 4.2 VSP/AIP 辅助数据模块

文档对 VSP 的定位很清晰：它服务前端展示、辅助标注和 GeoJSON 输出，但数据清洗、标准化和建模方式由 A-5 负责。

### API 设计权判断

以下接口应由 A-5 单方主导定义：

- 航路点查询
- 程序查询
- 机场基础信息查询
- 航司名称与编号映射查询
- GeoJSON 输出

### 表设计权判断

以下表应视为 A-5 完全主控：

- `vsp_airlines`
- `vsp_waypoints`
- `vsp_procedures`
- `vsp_airports`
- 视需要增加的 `vsp_runways`、`vsp_procedure_points`

### 结论

VSP 的原始资料来自外部，但“系统内如何组织成稳定可查的数据模型”完全属于 A-5 的设计权，因此它是第二个核心自主模块。

## 4.3 消息中间件与系统支撑模块

文档已明确 Redis 由 A-5 负责运行维护和消息生产消费接口。这里真正属于 A-5 的核心不是“装 Redis”，而是平台级约定。

### API/事件设计权判断

以下内容应由 A-5 主导：

- 队列名称规范
- 消息 JSON 结构
- `version`/`trace_id`/`producer` 等公共字段
- 消费失败重试策略
- 死信记录方式
- 队列监控指标

### 表设计权判断

以下支撑表适合由 A-5 自主设计：

- `event_dead_letters`
- `event_consume_failures`
- `system_logs`
- `system_configs`

### 结论

消息内容里的业务 `payload` 可能来自其他模块，但“消息契约外壳”和“基础中间件治理”属于 A-5 主控。

## 5. 半自主/需协商模块清单

## 5.1 航迹接入

`adsb_tracks` 虽然由 A-5 落库，但其业务来源是 A-1，字段含义还受公开 ADS-B 数据源影响。

### A-5 可以主导的部分

- 接口接收方式
- 入库校验方式
- 索引设计
- 时间与坐标统一格式
- 与其他表的关联键策略

### 需要协商的部分

- 呼号格式
- 轨迹唯一标识规则
- 原始字段保留策略
- 是否保留 source/raw_payload 字段

### 结论

这部分应定义“最小兼容 schema”，不宜在第一轮就把所有业务规则拍死。

## 5.2 语音元数据与任务配置

数据库设计说明书中已经明确把 `a2_voice_info`、`a2_task_realtime_cfg`、`a2_task_download_cfg` 归到 A-2 主导，这一点和“全部由 A-5 建模”的说法并不一致。

### A-5 可以主导的部分

- 接收接口规范
- 持久化字段映射
- 文件路径存储策略
- 索引、时间字段标准化

### 需要协商的部分

- `uniqueId` 生成规则
- 实时/历史任务状态码定义
- 采集与下载配置字段是否稳定
- 文件命名与目录组织规则

### 结论

`a2_*` 表不建议当作 A-5 完全自主表来设计。

## 5.3 ASR 与标注相关表

现有文档有一处容易误导的地方：数据库设计说明书把 `asr_results`、`annotation_tasks`、`annotation_results` 放进了数据库设计范围，甚至写成 “A-5” 章节的一部分，但从职责来源看，这些表并不是纯 A-5 业务。

### A-5 可以主导的部分

- 存储容器和关系约束
- 查询接口和保存接口外壳
- ID、时间、审计字段规范

### 需要协商的部分

- VAD 片段结构
- transcript 与 corrected_text 的粒度
- 标注状态流转
- 时间戳修正格式
- 任务分配逻辑

### 结论

这些表更像跨模块业务表。A-5 可承载，但不适合单方定义最终业务模型。

## 6. 明显不应由 A-5 单方拍板的模块清单

以下对象建议明确标注为“外部/他模块主导，A-5 仅兼容接入”：

- `adsb_tracks` 的完整业务字段集合
- `a2_voice_info`
- `a2_task_realtime_cfg`
- `a2_task_download_cfg`
- `asr_results`
- `annotation_tasks`
- `annotation_results`

原因有三点：

- 数据生产权不在 A-5
- 业务语义主要由对应模块决定
- 文档中已经出现跨模块归属痕迹，强行归为 A-5 会导致后续联调冲突

## 7. API 主控权分析

建议按“接口所有权”而不是按“接口是否由 A-5 提供”来判断。

### 7.1 A-5 完全主控 API

- `/api/v1/auth/login`
- `/api/v1/auth/refresh`
- `/api/v1/auth/logout`
- `/api/v1/users/me`
- `/api/v1/vsp/waypoints`
- `/api/v1/vsp/procedures`
- `/api/v1/vsp/airports`
- `/api/v1/vsp/airlines`
- `/api/v1/vsp/geojson/...`
- `/api/v1/system/queues`
- `/api/v1/system/events/failures`

### 7.2 A-5 主导承接，但需协商的 API

- `/api/v1/tracks/ingest`
- `/api/v1/audio/metadata`
- `/api/v1/asr/results`
- `/api/v1/annotations/load`
- `/api/v1/annotations/save`

### 7.3 设计建议

- 对完全主控 API，直接冻结请求/响应模型
- 对协商 API，先冻结最小字段集和版本号，不冻结所有业务细节
- 对所有外部接口统一加版本前缀 `/api/v1`

## 8. 数据表主控权分析

## 8.1 完全主控表

- `users`
- `user_login_audit`
- `user_refresh_tokens`
- `vsp_airlines`
- `vsp_waypoints`
- `vsp_procedures`
- `vsp_airports`
- `event_dead_letters`
- `event_consume_failures`
- `system_logs`

这些表的共同特征是：业务语义由 A-5 自己定义，字段变化不会直接破坏他模块核心逻辑。

## 8.2 协商接入表

- `adsb_tracks`
- `a2_voice_track_rel`
- `asr_results`
- `annotation_tasks`
- `annotation_results`

这些表可以由 A-5 建库和维护索引，但字段字典必须经跨模块确认。

## 8.3 他方主导表

- `a2_voice_info`
- `a2_task_realtime_cfg`
- `a2_task_download_cfg`

这些表即使最终和 A-5 数据库放在一起，也不建议由 A-5 单方定版。

## 9. 风险与文档冲突点

现有文档并不完全一致，至少有以下风险需要在后续实施前纠偏：

### 9.1 “A-5 负责数据库” 不等于 “A-5 拥有所有业务表设计权”

数据库设计说明书覆盖了系统核心业务表，但这更像“统一存储范围”，不是“统一业务所有权”。

### 9.2 SQLite 选型与字段写法存在冲突

文档中一边强调 SQLite，一边又出现：

- `POINT`
- `JSONB`
- PostgreSQL 风格空间字段表达

因此后续真正实现时，必须优先采用 SQLite 兼容设计，必要时把空间能力降级为 `lat/lng` 或文本化 GeoJSON。

### 9.3 标注相关表被写成 A-5 表，容易引发职责漂移

`annotation_tasks` 与 `annotation_results` 的业务核心其实服务于 A-4 标注流程。A-5 若单方定义状态流，会直接影响前端逻辑。

### 9.4 Redis 只是中间件，不等于消息语义已经定稿

文档已列出消息队列服务，但并未完整定义：

- 统一事件结构
- ack/重试策略
- 死信策略
- 监控口径

这些仍需要 A-5 统一补齐。

## 10. 建议实施顺序

为了让后续工作既能快速启动，又尽量减少返工，建议按以下顺序推进：

### 第 1 步：冻结对接矩阵

输出一份模块对接矩阵，明确：

- 谁生产数据
- 谁消费数据
- 谁拥有字段定义权
- A-5 负责接收、转发还是持久化

### 第 2 步：冻结 A-5 主控 canonical schema

优先定版以下表：

- `users`
- `user_login_audit`
- `vsp_airlines`
- `vsp_waypoints`
- `vsp_procedures`
- `vsp_airports`
- `event_dead_letters`

### 第 3 步：定义 Redis 事件契约

统一事件公共结构，至少包含：

- `id`
- `type`
- `version`
- `producer`
- `timestamp`
- `trace_id`
- `payload`

### 第 4 步：补齐 A-5 主控 API 契约

优先完成：

- 认证 API
- 用户信息 API
- VSP 查询 API
- GeoJSON API
- 队列监控/失败事件查询 API

### 第 5 步：再冻结协商型接口和表

对以下内容只先定义最小兼容字段集：

- 航迹接入
- 语音元数据接入
- ASR 结果接入
- 标注加载与保存

### 第 6 步：最后进入实现与联调

等对接边界稳定后，再进入：

- 迁移脚本
- FastAPI 接口
- Redis producer/consumer
- 联调测试

## 11. 审核建议

如果你是从“哪些内容我现在就能独立开工”这个角度审这份分析，那么优先级建议如下：

### 可以立即独立开工

- 用户与鉴权
- VSP/AIP 数据标准化建模
- Redis 统一事件壳与失败处理
- 审计/日志/死信表

### 需要先拿到接口共识再开工

- 航迹接入
- 语音元数据接入
- ASR 结果接入
- 标注任务与结果模型

### 不建议现在就写死

- A-2 任务配置细节
- annotation 状态流完整定义
- 依赖 PostgreSQL/SpatiaLite 特性的复杂字段

## 12. 本文档结论

按 A-5 主控口径，真正“无论 API 还是表设计都完全依赖于你自己”的，不是整个 A-5，而是 A-5 中的平台治理与支撑能力：

- 用户与鉴权
- VSP/AIP 支撑数据
- 消息中间件治理
- 系统支撑与审计类表

而航迹、语音、ASR、标注等跨模块业务对象，应当被视为“协商接入”或“他方主导、A-5 承接”，不宜直接作为 A-5 完全自主模块处理。
