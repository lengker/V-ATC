# Alpha A-5 自主可控模块分析与任务落地建议

## 1. 结论摘要

基于《需求规格说明书》《功能设计说明书》《可行性分析》《数据库设计说明书》以及现有 `docs/a5_db_middleware_plan.md`，按 **A-5 主控口径** 判断，A-5 范围内的内容可以分为三类：

1. **完全由 A-5 主控**
   - 用户与鉴权能力
   - VSP/AIP 辅助数据管理
   - 消息中间件封装、事件契约、失败处理与监控
   - A-5 自建的审计、日志、死信、消费失败等支撑表
2. **A-5 主导实现，但接口/字段需跨模块协商**
   - 航迹、语音、ASR、标注相关接入接口
   - 与 A-1 / A-2 / A-3 / A-4 的交换契约
   - 为跨模块数据落库而设计的兼容性表结构
3. **不应由 A-5 单方拍板**
   - A-1 主导的航迹业务字段
   - A-2 主导的语音采集任务与语音元数据字段
   - A-3 主导的 ASR / VAD 输出字段
   - A-4 主导的标注交互细节、标注任务流转字段

换句话说，A-5 的真正“自主权”主要集中在 **平台底座与支撑能力**，而不是所有业务表都由 A-5 一次性拍板。

## 2. 判断口径

本分析中的“完全依赖于我自己”不是按个人分工，而是按 **A-5 是否拥有最终定义权** 判断。为避免混淆，文中将主控权拆成三层：

- **模块主控权**：该能力是否属于 A-5 的核心职责。
- **API 主控权**：接口路径、鉴权方式、请求响应模型是否由 A-5 可以独立定义。
- **表设计主控权**：表名、字段、约束、索引是否可以由 A-5 单方确定。

只要有一层明显依赖其他模块，就不归为“完全自主”。

## 3. A-5 职责边界总览

从 4 份文档的一致表述看，A-5 的一级职责固定为以下五块：

- 数据库运维与管理模块（A5-DB）
- 跨模块 API 接口服务模块（A5-API）
- 用户管理与鉴权模块（USR）
- VSP/AIP 数据管理模块（VSP）
- 消息中间件服务模块（MSG）

其中真正拥有较强自主定义权的，不是整个 A5-DB 或整个 A5-API，而是其中与 **用户、VSP、消息治理、支撑表** 有关的部分。

## 5. 完全自主模块清单

### 5.1 用户与鉴权

这一部分是最明确的 A-5 主控域。

- A-5 可以独立定义登录、Token 刷新、注销、鉴权拦截策略。
- A-5 可以独立定义 `users` 表的主键、用户名、密码哈希、角色、状态、最后登录时间等字段。
- A-5 还可以自主决定第一版是否采用简化角色模型：
  - 先在 `users.role` 里放 `admin` / `annotator`
  - 后续再按需要拆成 `roles` / `permissions` / `user_roles`

这一块对其他模块的影响主要是“消费你的鉴权结果”，而不是“决定你的表结构”。

### 5.2 VSP/AIP 辅助数据

这一部分也是 A-5 的核心主控域。

- 数据原始来源虽然是 AIP、VSP、公开航空资料，但系统内如何抽取、标准化、落成查询结构，由 A-5 决定。
- A-5 可以自行确定 `vsp_airlines`、`vsp_waypoints`、`vsp_procedures`、`vsp_airports` 等表的组织方式。
- A-5 可以自行决定对外查询接口和 GeoJSON 输出结构，只需满足 A-4 可消费即可。

这类表的业务含义主要是“辅助展示与辅助标注”，不是其他模块私有业务的直接镜像，因此最适合由 A-5 统一治理。

### 5.3 消息中间件与事件治理

文档中把 Redis 环境、消息队列定义、生产消费接口都归在 A-5，因此这一块也应视为 A-5 完全主控。

- 队列命名规则由 A-5 主导。
- 统一消息结构由 A-5 主导，例如：
  - `id`
  - `type`
  - `payload`
  - `timestamp`
  - `producer`
  - `version`
  - `trace_id`
- 失败重试、死信记录、消费失败日志、监控指标由 A-5 主导。

即使具体 `payload` 内部字段可能来自其他模块业务，也不影响 **消息契约外壳、治理策略、可靠性机制** 由 A-5 拍板。

### 5.4 A-5 自建支撑表

下列表更适合纳入 A-5 的自主范围：

- `user_login_audit`
- `event_consume_failures`
- `event_dead_letters`
- `system_operation_logs`
- 其他不承载外部业务真相、只服务于平台治理的审计类表

这些表不与 A-1~A-4 的业务真相冲突，完全可以由 A-5 自主设计。

## 6. 半自主 / 需协商模块清单

### 6.1 跨模块接入 API

A-5 可以主导“接口服务层的实现方式”，但不能单方定义所有字段。

具体表现为：

- **航迹接入接口**
  - A-5 可以决定是否通过 HTTP / 消息消费 / 批量导入方式接入
  - 但 `callsign`、坐标、速度、高度、时间戳等字段语义应以 A-1 为准
- **语音元数据接口**
  - A-5 可以决定接口路径、分页格式、响应包装
  - 但录音主键、频段、时间、文件路径、任务状态等字段需要 A-2 确认
- **ASR 结果接口**
  - A-5 可以决定入库校验和统一响应
  - 但 `vad_segments`、`transcript`、`confidence`、`engine` 等字段含义应由 A-3 确认
- **标注加载 / 提交接口**
  - A-5 可以提供统一鉴权、保存、查询能力
  - 但页面加载形态、保存粒度、标注状态机需和 A-4 对齐

因此这些接口应归类为 **A-5 主导实现，双方共同冻结契约**。

### 6.2 兼容性数据表

只要某张表本质上是在承接别的模块产生的业务真相，它就不应被算作 A-5 完全自主表。

原因有两个：

- 数据字段由生产方定义，A-5 只是存储与查询承接方。
- 下游行为依赖生产方语义，如果 A-5 单方修改字段，联调会立即失效。

所以 A-5 更合适的职责是：

- 定义最小可接入字段集合
- 定义索引、唯一约束、兼容字段映射
- 定义迁移和版本化策略
- 避免替其他模块过早细化业务含义

## 7. 明显不应由 A-5 单方拍板的对象

### 7.1 `adsb_tracks`

虽然该表会落在 A-5 管理的数据库里，但字段语义本质属于 A-1 的航迹处理能力。

- 数据来源：A-1
- 业务语义归属：A-1
- A-5 合理职责：接收、存储、索引、查询兼容

因此 `adsb_tracks` 不应认定为 A-5 完全自主表。

### 7.2 `a2_voice_info` 与 `a2_task_*`

数据库说明书中已经明确把它们归在 “A-2 语音模块任务与配置表”。

- 语音唯一标识如何生成
- 实时采集任务参数如何定义
- 历史下载任务调度字段如何组织

这些都更像 A-2 内部业务模型，而不是 A-5 平台模型。

### 7.3 `asr_results`

需求和数据库设计都表明 A-3 负责识别结果输出。

A-5 可以：

- 设计入库接口
- 加索引
- 做版本兼容

但不应单方决定：

- `vad_segments` 的格式
- `confidence` 的含义
- `engine` 的标识规则
- 一条录音是否允许多版本识别结果并存

### 7.4 `annotation_tasks` 与 `annotation_results`

这两张表虽然在文档里被写进数据库设计，但主业务压力来自 A-4 标注界面。

真正影响字段设计的是：

- 页面一次加载什么数据
- 用户一次保存哪些内容
- 标注状态流转有哪些节点
- 是否支持多人协同、回退、审核

这些都不是 A-5 能脱离 A-4 单独决定的。因此它们更适合被归为 **协商设计表**。

## 8. API 主控权分析

### 8.1 可由 A-5 独立冻结的 API

这类接口可以先行定稿：

- `/api/v1/auth/login`
- `/api/v1/auth/refresh`
- `/api/v1/auth/logout`
- `/api/v1/users/me`
- `/api/v1/vsp/waypoints`
- `/api/v1/vsp/procedures`
- `/api/v1/vsp/airports`
- `/api/v1/vsp/geojson`
- `/api/v1/system/queue-status`
- `/api/v1/system/events/failures`

原因是这些接口的请求响应模型主要由 A-5 内部能力决定。

### 8.2 需要先冻结契约再实现的 API

这类接口不建议 A-5 单方写死字段：

- `/api/v1/tracks/ingest`
- `/api/v1/audio/metadata`
- `/api/v1/asr/results`
- `/api/v1/annotations/tasks`
- `/api/v1/annotations/results`

这类 API 的正确做法不是先实现细节，而是先产出：

- 模块对接矩阵
- 最小字段集合
- 版本号策略
- 错误码和幂等规则

## 9. 数据表主控权分析

### 9.1 完全自主表

建议直接视为 A-5 canonical schema 的核心：

- `users`
- `user_login_audit`
- `vsp_airlines`
- `vsp_waypoints`
- `vsp_procedures`
- `vsp_airports`
- `event_consume_failures`
- `event_dead_letters`
- `system_operation_logs`

这些表的字段可以直接由 A-5 冻结并进入迁移脚本。

### 9.2 兼容接入表

建议先定义“最小兼容模型”，不要过度细化：

- `adsb_tracks`
- `a2_voice_info`
- `a2_voice_track_rel`
- `a2_task_realtime_cfg`
- `a2_task_download_cfg`
- `asr_results`
- `annotation_tasks`
- `annotation_results`

对这类表，A-5 应主导的是：

- 命名规范
- 时间字段统一规范
- 索引
- 版本化迁移
- SQLite 可落地的数据类型替换策略

而不是字段业务含义本身。

## 10. 风险与文档冲突点

### 10.1 “A-5 管数据库”不等于 “A-5 拥有所有表定义权”

这是当前文档最容易造成误解的地方。

- 需求与功能文档把很多表放进 A-5 范围，是因为数据库由 A-5 管。
- 但数据库说明书同时又写明：
  - A-1 汇入航迹
  - A-2 主导语音元信息和任务配置
  - A-3 / A-4 涉及识别与标注业务表
  - A-5 负责用户与 VSP 支撑表

因此应把“数据库托管权”和“业务定义权”分开。

### 10.2 SQLite 选型与字段写法存在冲突

已有文档中出现了 `POINT`、`JSONB`、`NOW()` 等更偏 PostgreSQL/PostGIS 风格的写法，而项目又明确选型 SQLite。

这意味着：

- A-5 不能直接照抄数据库说明书落库
- 需要把空间字段改写为 `lat` / `lng` 或可选 SpatiaLite 方案
- 需要把 JSON 相关能力按 SQLite 可落地方式处理

这进一步说明，A-5 当前更应该先输出 **canonical schema 与兼容策略**，而不是仓促实现所有表。

### 10.3 标注表在文档中被写成 “A-5 业务表”，但业务真实主导方并不是 A-5

`annotation_tasks`、`annotation_results` 被写进数据库设计，不等于 A-5 能完全拍板。

因为标注流本质依赖：

- A-4 的前端交互
- A-3 识别结果粒度
- 用户角色和状态设计

所以这类表必须保留协商空间。

### 10.4 Redis 队列可靠性描述仍需细化

文档只说了 Redis、消息队列、生产消费接口，但没有完整冻结：

- ack 策略
- 重试次数
- 幂等键
- 死信机制
- 监控指标

这一部分正好是 A-5 可以补强并自主定义的内容，应优先补成正式契约。

## 11. 建议实施顺序

### 阶段 1：冻结对接矩阵

先明确 A-1 到 A-4 与 A-5 的边界：

- 谁生产数据
- 谁消费数据
- 谁拥有字段定义权
- 谁负责兼容历史版本

产物建议：

- 模块对接矩阵
- 接口清单
- 事件清单
- 兼容性说明

### 阶段 2：冻结 A-5 主控 canonical schema

优先冻结完全自主表：

- `users`
- `vsp_*`
- 审计 / 死信 / 日志类表

这些表可以直接形成第一版迁移基线。

### 阶段 3：定义 Redis 事件契约

输出统一消息规范：

- 队列名
- 消息结构
- 版本号
- 重试规则
- 死信策略
- 监控字段

### 阶段 4：定义 API 契约

实施顺序建议为：

1. 鉴权接口
2. VSP 查询接口
3. 队列监控与系统接口
4. 航迹 / 语音 / ASR / 标注接入接口

### 阶段 5：进入实现与联调

联调闭环建议按最小可用路径推进：

1. A-4 登录获取 Token
2. A-4 查询 VSP 数据
3. A-1 或模拟端投递航迹
4. A-3 或模拟端提交 ASR 结果
5. A-4 拉取并保存标注

## 12. 给当前工作的明确建议

如果后续要正式进入实现，最稳妥的顺序不是“先把所有表建出来”，而是：

1. 先固化 A-5 自主域和协商域
2. 再冻结 A-5 主控表结构
3. 再补消息契约
4. 再补 API 契约
5. 最后实施数据库、接口和联调

这样做的收益是：

- 避免把文档中的冲突直接固化为代码
- 让其他模块尽早知道哪些字段不能随意改
- 让 A-5 的产物更像平台底座，而不是一次性对接脚本
- 后续联调时表结构调整成本更低
