# Alpha A-5 数据库与消息中间件实施计划

## 1. 你的实际负责范围

你负责的是 A-5 模块里最适合独立落地、且最需要和其他模块对接的两部分：

- 数据库环境与数据库对接层
- 用户鉴权相关表与接口
- VSP/AIP 辅助数据相关表与接口
- 消息中间件设计、封装与监控
- 与 A-1/A-2/A-3/A-4 的数据交换契约

不建议你直接“拍板”所有业务表结构。根据文档内容，以下表更像是其他模块主导、A-5 负责兼容接入：

- `adsb_tracks`
- `a2_voice_info`
- `a2_task_realtime_cfg`
- `a2_task_download_cfg`
- `asr_results`
- `annotation_tasks`
- `annotation_results`

处理原则：

- 这些表先定义“接入契约”和最小兼容字段，不先写死实现细节。
- 真正由你主导的数据表，优先限定在 `users`、`vsp_*`、消息事件日志与必要的通用支撑表。

## 2. 文档里已经确认的硬约束

- 后端框架：FastAPI
- 数据库：SQLite，文档倾向 `SQLite + SpatiaLite`
- 消息中间件：Redis
- 鉴权：JWT
- 文档接口规范：OpenAPI 3.1 / Swagger
- 地图辅助数据输出：GeoJSON
- 目标机场：`VHHH`
- 运行环境：Debian x64，`1 vCPU / 512 MB / 10 GB`

## 3. 需要先纠偏的文档问题

这些问题说明你不能直接照着两份数据库说明书建表：

1. 文档一边写“SQLite”，一边又出现 `SERIAL`、`PostGIS POINT` 这类 PostgreSQL 风格字段。
2. `vsp_procedures.waypoints` 在一份文档里甚至被写成了 `INTEGER PRIMARY KEY AUTOINCREMENT`，明显不合理。
3. 文档前半段像是“全库由 A-5 设计”，后半段又明确说“非用户相关、非 VSP 相关表结构遵循其他模块要求并由其他模块主导”。
4. Redis 队列写了“支持确认机制”，但又指定使用 Redis List；如果要强可靠确认，后续需要自己补消费状态或改成 Streams。

因此后续产物应采用“可替换、可修订”的组织方式：

- SQL 用迁移脚本管理
- API 用版本化路径管理，如 `/api/v1/...`
- Redis 消息体定义为单独文档
- 对外字段尽量稳定，对内实现允许替换

## 4. 建议你按这个顺序完成

### 第 1 步：先冻结对接边界，而不是先建表

先输出一份 A-5 对接清单，明确每个模块和你之间谁生产数据、谁消费数据、谁拥有字段定义权。

- A-1 -> A-5：航迹入库消息或接口
- A-2 -> A-5：语音元数据、录音路径、时间信息
- A-3 -> A-5：ASR 结果、VAD 片段、引擎版本
- A-4 -> A-5：登录、用户信息、VSP 查询、标注保存、标注通知

这一阶段的交付物：

- 模块对接矩阵
- 外部表字段最小集合
- 事件清单
- API 清单

### 第 2 步：确定你的“主控数据库模型”

建议你把数据库分成三层：

- 主控表：你自己负责定义与维护
- 外部业务表：你只做兼容，不做业务裁决
- 派生或缓存表：为查询性能和中间状态服务

建议优先落地的主控表：

- `users`
- `user_login_audit`
- `vsp_airlines`
- `vsp_waypoints`
- `vsp_procedures`
- `vsp_airports`
- `event_dead_letters` 或 `event_consume_failures`

建议保留但不要一开始做复杂化的内容：

- `roles`
- `permissions`
- `user_roles`

如果当前只有 `admin` 和 `annotator` 两种角色，第一版可先在 `users.role` 落地，等需要扩展时再拆 RBAC 表。

### 第 3 步：先定义 canonical schema，再写迁移

你真正需要产出的不是“最终表格截图”，而是“可反复修改的数据库定义”：

- `schema.sql` 或迁移脚本目录
- 初始化脚本
- 种子数据脚本
- 字段字典文档

SQLite 下建议的数据设计原则：

- 时间统一存 ISO 8601 UTC 字符串
- 地理坐标先用 `lat` + `lng` 落地
- GeoJSON 在接口层生成，不强依赖库内空间类型
- SpatiaLite 作为可选增强，不作为第一阶段阻塞项

这一步的目的，是保证在 Debian 低配置机器上也能先跑通。

### 第 4 步：设计消息中间件，不要只停留在“装 Redis”

你需要定义 4 类东西：

- 队列名
- 消息 JSON 格式
- 生产/消费责任
- 失败重试和死信策略

文档里已经给出的候选队列：

- `track:ingest`
- `audio:process`
- `annotation:notify`
- `system:log`

建议你补齐统一消息结构：

- `id`
- `type`
- `payload`
- `timestamp`
- `producer`
- `version`
- `trace_id`

建议第一版实现策略：

- 生产者统一走一个消息发布服务
- 消费者统一走一个消费封装层
- 失败消息写入死信表或死信队列
- 队列监控至少提供长度、失败次数、最近消费时间

### 第 5 步：优先做“别人马上要接”的接口

接口优先级建议如下：

第一优先级：

- 登录
- Token 刷新
- 用户信息查询
- VSP 航路点查询
- VSP 程序查询
- GeoJSON 输出

第二优先级：

- 航迹接收接口或消息消费入库
- 语音元数据接收接口
- ASR 结果接收接口
- 标注结果保存接口

第三优先级：

- 管理接口
- 数据导出
- 队列监控接口

### 第 6 步：给接口和消息都加“版本意识”

为了后续方便改和对接，建议统一做以下约束：

- API 路径全部使用 `/api/v1/...`
- Redis 消息体加入 `version`
- 所有外部入参用 Pydantic 模型
- 所有外部响应固定包一层响应模型
- 所有表结构改动必须通过迁移脚本

### 第 7 步：先完成最小联调闭环

你的最小闭环不应该是“所有表都建完”，而应该是：

1. A-4 能登录拿 Token
2. A-4 能查到 VSP 数据
3. A-1 或模拟端能投递一条航迹消息
4. A-5 能消费消息并持久化
5. A-3 或模拟端能提交一条 ASR 结果
6. A-4 能读取并展示结果

只要这条链路跑通，你的部分就已经具备联调价值。

## 5. 推荐的最终交付物

为了便于修改、便于对接，你的工作成品建议至少包括：

- `docs/` 下的数据库字段字典
- `docs/` 下的消息事件契约文档
- `docs/` 下的 API 对接文档
- 数据库迁移脚本
- 初始化脚本
- Redis 配置样例
- `.env.example`
- 最小联调测试清单

## 6. 我建议你实际执行时的任务拆分

### 阶段 A：需求定稿

- 从 4 份文档提炼 A-5 真实职责
- 把“自建设计”与“外部模块主导”拆开
- 输出数据库与消息对接矩阵

### 阶段 B：数据库底座

- 选定 SQLite 第一版目录结构
- 定义主控表
- 编写迁移脚本和初始化脚本
- 准备种子数据

### 阶段 C：消息中间件

- 安装并配置 Redis
- 定义队列名和消息格式
- 编写统一 producer / consumer 封装
- 完成失败重试和监控

### 阶段 D：接口层

- 完成认证接口
- 完成 VSP 查询接口
- 完成跨模块接入接口或消息消费服务
- 生成 Swagger 文档

### 阶段 E：联调与验收

- 使用模拟数据联通 A-1/A-3/A-4
- 验证数据库写入、查询、鉴权、消息消费
- 补充异常场景和回归测试

## 7. 我给你的明确建议

如果你现在要开始正式做，最合理的顺序不是“先把数据库设计说明书补完”，而是：

1. 先写 A-5 对接矩阵
2. 再定主控表的 canonical schema
3. 再定 Redis 事件契约
4. 再做 FastAPI 的接口壳子
5. 最后才补完整实现和联调

这样做的好处是：

- 后续表结构改动成本低
- 别的模块能更早和你联调
- 不会把错误文档直接落成代码
- 你的产物更像“平台能力”，而不是一次性课程作业代码
