# 项目改进工作总结

时间范围：2026-05-27 至 2026-06-02  
涉及模块：A-2 语音处理模块、A-1 ADS-B 航迹模块、前端联调页面、Alpha ASR/元数据集成

 一、总体工作概述

本阶段主要围绕 ATC 语音下载、LiveATC 接入、实时/历史数据联调、ASR 识别链路、A1 航迹联动以及前端交互稳定性开展改进。工作重点不是单一页面调整，而是贯穿了前端、A-2 后端、A-1 航迹接口、Alpha 元数据与 ASR 服务之间的完整链路。

本周完成的核心改进包括：

- 修复 LiveATC 历史下载导入中 Cloudflare 安全校验导致的下载失败问题。
- 区分并恢复“实时下载”和“下载导入”的不同业务逻辑，避免两条链路互相污染。
- 优化 LiveATC 归档下载流程，使其能够在勾选安全校验后继续执行 Submit、解析音频地址并完成下载入库。
- 将下载导入后的语音记录同步到 Alpha，并修复 ASR 使用错误音频地址导致的识别失败问题。
- 在下载导入成功后联动 A1 启动 ADS-B 航迹抓取任务，实现语音与航迹的协同采集。
- 修复录音列表切换异常，使导入后的临时音频状态不会阻塞用户切换其它录音。
- 梳理 A1 模块现有航迹能力，明确其实时抓取、本地查询、航线重建与“指定时间范围历史下载”的边界。

 二、A-2 LiveATC 下载导入链路改进

# 1. Cloudflare 安全校验处理

LiveATC 归档页面在访问过程中会出现 Cloudflare Turnstile 安全校验，原逻辑只做了一层通用绕过检测，无法覆盖归档页内部出现的第二个复选框。

本阶段针对该问题补充了 SeleniumBase 手动点击逻辑：

- 检测 Cloudflare challenge iframe 是否存在。
- 切入 iframe 查找 `input[type="checkbox"]` 或 label。
- 自动点击可见复选框。
- 点击后保留等待时间，确保页面完成验证后再进入后续逻辑。
- 避免过度依赖 `#success` 状态，因为部分 Turnstile iframe 内部状态不稳定，可能造成代码卡在等待中。

改进效果：

- 下载导入流程可以正常通过安全检测。
- 安全检测完成后可以继续点击 Submit。
- 避免前端过早反馈下载失败但浏览器后台仍在继续执行的问题。

# 2. Submit 后音频结果解析

针对 Submit 后页面跳转到音频播放条但后端仍无法下载的问题，补充并调整了归档结果处理逻辑：

- 等待 `#archiveResults` 出现并加载内容。
- 识别错误文本，例如 `Error retrieving archive`、`HTTP 403`。
- 支持从 `audio`、`source`、`a[href]` 中提取 `.mp3` 音频地址。
- 对需要先点击音频链接才能出现播放器的场景，补充点击归档结果链接后的等待。
- 使用浏览器会话中的 Cookie、User-Agent、Referer 下载真实音频地址，减少 403 风险。

改进效果：

- LiveATC 历史归档下载从“只能打开页面”推进到“可以获取音频 URL、下载、归一化并入库”。
- 对归档页延迟加载、链接跳转、播放器生成等情况的兼容性更强。

# 3. 实时下载与下载导入逻辑分离

调试过程中发现实时下载和历史下载导入不能复用同一套浏览器下载逻辑。实时下载本身应通过 LiveATC 直播流地址或 ASX 解析得到实时流，而不是跳出浏览器归档页面。

本阶段完成了逻辑回退和分离：

- 实时下载保持原有通过流地址接收分段音频的模式。
- 下载导入继续使用 SeleniumBase 访问 LiveATC 归档页。
- 避免实时下载按钮触发 Chrome 页面或归档页流程。
- 保持 A-2 `runtime_service.py` 中实时接收的直接流式读取逻辑。

改进效果：

- 实时下载和历史下载导入两条链路职责清晰。
- 降低了后续维护时误改另一条链路的风险。

 三、前端下载导入与 Alpha 集成改进

# 1. 下载导入前端超时与代理问题处理

在联调中出现 Next.js 代理请求 `socket hang up` 的情况，表现为前端已经显示下载失败，但 Selenium 浏览器窗口仍在执行 Submit 和页面跳转。

本阶段将 A-2 API 请求从 Next 代理链路调整为直接访问 A-2 服务地址：

- 使用 `NEXT_PUBLIC_A2_API_BASE_URL` 或默认 `http://127.0.0.1:8001`。
- 避免长耗时 Selenium 下载过程被前端代理连接中断。

改进效果：

- 减少下载导入过程中前端提前报错的情况。
- A-2 长耗时下载任务更适合由浏览器直接请求后端服务。

# 2. Alpha 元数据同步

下载导入完成后，前端会把 A-2 返回的语音记录转成 Alpha 可识别的音频元数据，并同步到 Alpha：

- 保存 `unique_id`、ICAO、频段、开始时间、结束时间等字段。
- 将 A-2 可播放音频地址作为 Alpha 的 `file_path`。
- 触发录音列表刷新和当前录音选中。

改进效果：

- 下载导入后的语音可以出现在统一录音列表中。
- 前端主页面可以直接加载刚下载的真实录音。

# 3. ASR 识别失败修复

下载导入后的语音曾出现 ASR 识别失败。原因是前端给 ASR 使用的是 A-2 原始文件接口 `/api/a2/voice/file/{id}`，而播放器使用的是 `/playable` 转码后的 WAV 地址。原始 MP3 可能存在浏览器或 ASR 后端解析不稳定的问题。

本阶段修复为：

- 录音加载时统一使用 `/api/a2/voice/file/{id}/playable`。
- ASR 输入地址也使用 `/playable`。
- 同步到 Alpha 的文件名改为 `${unique_id}.wav`，与实际可播放 WAV 内容保持一致。

改进效果：

- ASR 获取音频的格式更稳定。
- 避免文件扩展名与实际音频内容不一致造成识别失败。

 四、A1 ADS-B 航迹联动改进

# 1. 下载导入后同步启动航迹抓取

此前下载导入只下载语音，没有同时准备航迹数据。为了和实时下载体验保持一致，本阶段在下载导入成功后增加 A1 航迹任务启动逻辑：

- 生成与当前 ICAO、频段、语音记录相关的 A1 taskId。
- 调用 A1 `POST /api/adsb/routes/crawl-tasks/start`。
- 使用 `airplanes-live` provider 和香港/VHHH preset。
- 将 A1 taskId 展示在下载导入完成提示中。
- A1 启动失败时只提示 ADS-B 失败，不影响语音下载导入成功状态。

改进效果：

- 下载语音的同时开始准备 ADS-B 航迹。
- 语音和航迹两个模块的协同程度提高。
- 避免 A1 临时异常导致 A-2 语音导入被误判失败。

# 2. A1 能力边界梳理

本阶段还对 A1 模块进行了代码级检查，明确当前 A1 具备以下能力：

- 实时从 Airplanes.live 或 OpenSky 拉取 ADS-B 状态点并写入 `adsb_tracks`。
- 启动后台定时爬取任务。
- 查询本地已有航迹点，支持 `start_time`、`end_time`。
- 从本地已有航迹点重建 `adsb_routes`。

同时明确当前 A1 尚不具备完整的“按指定时间范围去公网下载历史航迹”的能力。OpenSky 支持单个 `time` 快照参数，但项目中还没有实现从 `start_time` 到 `end_time` 按间隔循环抓取并落库的历史爬虫。

该梳理为后续开发 A1 历史航迹下载任务提供了明确方向。

 五、录音列表与页面交互修复

下载导入完成后，页面会保存一个临时 `loadedA2Audio` 状态。后续点击录音列表中其它录音时，原逻辑只更新 URL 参数，没有及时清理临时音频状态，导致用户看到录音列表无法正常切换。

本阶段修复了录音选择逻辑：

- 新增统一的录音选择处理函数。
- 点击其它录音时清理不匹配的临时 A2 音频。
- 优先使用列表缓存即时切换当前 active 音频。
- 再通过 URL 参数触发后端加载完整录音详情、ASR 和航迹数据。

改进效果：

- 录音列表不同录音之间可以正常切换。
- 导入后的临时音频不会把页面状态“钉住”。
- 用户操作反馈更及时。
